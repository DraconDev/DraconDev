# DraconDev Non-Employment Business Positioning Strategy

**Date:** 2026-06-12  
**Purpose:** Move DraconDev from employment-style income toward repeatable product, tool, license, and asset revenue.

## Core answer

The best positioning is:

> **DraconDev is a maker of usable automation products: Chrome extensions, tools, games, and Rust infrastructure that people can install, use, or license.**

The business should not be positioned as employment, freelancing, teaching, or generic open-source maintenance. GitHub is credibility and distribution. `dracon.uk` is the product destination. YouTube is acquisition. Paid products are the money path.

The highest-leverage strategy is:

1. **Productize one browser-extension wedge first.**
2. **Route all product traffic to `dracon.uk` or verified install/play pages.**
3. **Use GitHub only to prove the maker can build serious systems.**
4. **Use YouTube to demo the product, not to sell employment or courses.**
5. **Keep build-with foundations secondary unless they become productized assets.**

## Strategic north star

DraconDev should become a portfolio of repeatable products and assets, not a resume for hired work.

When complete, the business position should make these facts obvious to a visitor:

1. DraconDev makes usable things people can install, open, play, or license.
2. The main products are Chrome extensions, tools, and eventually platforms.
3. Games can exist, but they are not the primary business wedge unless a public play/install page is verified.
4. Build-with foundations are supporting assets, not the main offer.
5. Code is proof of capability, not the primary sales path.
6. Revenue comes from products, licenses, subscriptions, paid tiers, or sponsor support, not hourly employment.

## Positioning statement

Use this as the business-positioning spine:

> **I build usable automation products for the browser, terminal, creator workflow, and small infrastructure teams. Inspect the code if you want; use or buy the finished product when it is ready.**

Shorter profile version:

> **I make usable things people can run or play. Chrome extensions, games, tools, and Rust infrastructure.**

Business version:

> **DraconDev turns shipped tools into productized assets: browser extensions, creator tools, developer infrastructure, and small-team automation.**

Avoid positioning that implies employment-based income:

- “Available for work.”
- “Freelance developer.”
- “I teach developers.”
- “Hire me to build your app.”
- “Custom software consulting” unless it is packaged as a fixed-scope productized offer.
- “Learning journey” or “follow my tutorials” unless courses are intentionally the business.

## Target customer segments

### 1. Browser-product users

**Who:** General browser users who want faster workflows, AI assistance, summarization, form filling, debugging, or browser automation.

**Best-fit products:** SamAI, vidpro-extension, api-debugger, auto-form-filler, bugkit.

**Revenue model:** Freemium, paid tier, one-time purchase, or BYOK plus hosted AI.

**Why this is high leverage:** Chrome extensions are easy to install, easy to demo, and easy to connect to YouTube content.

### 2. YouTubers and creator-tool users

**Who:** Creators who need faster titles, tags, descriptions, upload workflows, metadata optimization, or video processing.

**Best-fit products:** vidpro-extension, youtube-video-uploader, avid, video-factory.

**Revenue model:** Creator pro tier, paid hosted processing, or bundled creator-tool subscription.

**Why this is high leverage:** DraconDev already has a YouTube channel and can demo tools directly to a relevant audience.

### 3. Small infrastructure teams

**Who:** Developers, solo founders, and small teams managing 5–100 VPS instances, deploys, backups, or service health.

**Best-fit products:** pully-fully-pull-based-fleet-reconciler, dracon-platform, dracon-utilities, dracon-demons.

**Revenue model:** Open-source core plus paid managed hosting, paid enterprise features, or self-hosted license.

**Why this is high leverage:** The work is technically differentiated and can become a serious B2B product.

### 4. Dev teams and automation agencies

**Who:** Teams that need browser automation, QA capture, API debugging, or repeatable browser workflows.

**Best-fit products:** rust-ai-web-auto, bugkit, api-debugger, ai-auto-repo-rot-scanner-todo-agent.

**Revenue model:** Open-source core plus paid hosted service, team license, or enterprise features.

**Why this is high leverage:** Browser automation and QA capture are painful, recurring problems with clear business value.

### 5. Rust/TUI developers

**Who:** Developers building terminal apps, Rust tooling, or low-level developer utilities.

**Best-fit products:** dracon-terminal-engine, tiles-tui-file-manager, dracon-libs, folder-auto-banner.

**Revenue model:** Mostly credibility and distribution. Optional paid support, hosted services, or enterprise features only if productized.

**Why this is medium leverage:** Strong proof, but less direct revenue than browser extensions and creator tools.

### 6. Gamers

**Who:** Roguelike, survival, visual novel, or indie-game players.

**Best-fit products:** Junk-Runner-bevy, one-mil-girls.

**Revenue model:** Steam, itch.io, paid download, or public demo plus wishlist funnel.

**Why this is lower priority:** Games can monetize, but current public install/play destinations are not verified, and the strongest near-term business wedge is usable browser/tool products.

## Recommended offer ladder

### Tier 0: Free public proof

Purpose: trust and distribution.

- Public GitHub repos with clear one-line value.
- Verified Chrome Web Store install pages.
- Verified game play/install pages when available.
- YouTube demos.
- Short README/product sections.

Do not charge here. Use it to create trust and traffic.

### Tier 1: Free-to-install product

Purpose: acquisition.

Examples:

- Free Chrome extension tier.
- Free BYOK tier.
- Public game demo.
- Open-source core.

Success metric: installs, active users, activation, repeat use.

### Tier 2: Paid product tier

Purpose: first monetization.

Examples:

- SamAI paid hosted AI tier.
- vidpro creator pro tier.
- api-debugger team/pro tier.
- bugkit paid capture bundles.
- auto-form-filler paid AI workflow tier.

Success metric: paid conversion rate and monthly recurring revenue.

### Tier 3: Productized platform or license

Purpose: higher-value non-employment revenue.

Examples:

- dracon-platform self-hosted license.
- pully-fully managed fleet reconciliation.
- rust-ai-web-auto team/enterprise license.
- dracon-utilities paid self-hosted bundle when public and clean.

Success metric: license revenue, managed hosting revenue, support hours per customer, renewal rate.

### Tier 4: Sponsor/support layer

Purpose: optional recurring support, not the main business.

Examples:

- GitHub Sponsors.
- Ko-fi on `dracon.uk`.
- Affiliate links only when directly relevant.
- Paid video sponsors only when the channel has enough consistent traffic.

Success metric: sponsor/support revenue as a percentage of total revenue. Keep it secondary unless it becomes meaningful.

Choose one product to make the first serious monetization test.

**Why:** Browser extensions are the cleanest path from profile → install → use → paid tier. They fit the current documentation direction and can be demoed on YouTube.

**Required moves:**

- Verify the public install page.
- Add a clear product page on `dracon.uk`.
- Add pricing or a paid-tier path.
- Add a short demo video.

**Assumption to verify:** `PRODUCT_INVENTORY.md` says SamAI is already on the Chrome Web Store. Verify before making public claims.

### Priority 2: vidpro as the creator-tool wedge

**Why:** DraconDev already has a YouTube channel. A tool for YouTubers has natural distribution and product-market clarity.

**Offer:** “YouTube optimization tool for faster titles, tags, descriptions, and upload workflow.”

**Revenue model:** Free tier plus creator pro tier.

### Priority 3: pully-fully as the infrastructure product

**Why:** It solves a concrete small-team infrastructure problem and is technically strong.

**Offer:** “Git-driven fleet reconciliation for 5–100 VPS.”

**Revenue model:** Open-source core plus paid managed hosting or paid self-hosted license.

### Priority 4: rust-ai-web-auto as the developer automation product

**Why:** Browser automation is a business problem, and Rust-based AI automation is differentiated.

**Offer:** “Reliable browser automation for teams that need repeatable workflows.”

**Revenue model:** Open-source core plus paid hosted service or enterprise license.

### Priority 5: games as optional digital products

**Why:** Games can monetize, but only when public play/install pages exist.

**Offer:** Public demo → wishlist → Steam/itch release.

**Revenue model:** Paid download, DLC, or later bundle.

**Constraint:** Do not promote private games as public products until a public play/install destination is verified.

## Monetization paths

### Path A: Browser-extension freemium

Best for: SamAI, vidpro, api-debugger, auto-form-filler.

Structure:

1. Free install.
2. Clear usage limit or feature split.
3. Paid hosted AI or pro features.
4. Upgrade CTA inside the product.
5. YouTube demo drives installs.

This is the best near-term path because it is repeatable and not employment-based.

### Path B: Open-source core plus paid product

Best for: pully-fully, rust-ai-web-auto, dracon-terminal-engine, dracon-libs.

Structure:

1. Publish useful core.
2. Use GitHub for trust.
3. Sell hosted, managed, team, or enterprise features.
4. Keep support productized and bounded.

Avoid custom one-off work. Sell a repeatable product or license.

### Path C: Self-hosted license

Best for: dracon-platform, dracon-utilities, pully-fully, rust-ai-web-auto.

Structure:

1. Private or public code.
2. Clear license terms.
3. Paid self-hosted deployment.
4. Optional paid updates/support window.

This is stronger than hourly work because the buyer pays for the asset and deployment rights.

### Path D: Creator tools

Best for: vidpro, youtube-video-uploader, avid, video-factory.

Structure:

1. Demo the tool while using it.
2. Offer free utility.
3. Sell pro automation, hosted processing, or creator workflow features.

This path pairs naturally with YouTube as distribution.

### Path E: Sponsor/support

Best for: GitHub Sponsors, Ko-fi, affiliate links, paid sponsors.

Structure:

1. Keep the sponsor/support path visible but not dominant.
2. Use it to support product development.
3. Do not rely on it as the main business unless traffic justifies it.

Sponsors are useful, but the core business should remain productized assets.

## Profile and business funnel

### GitHub profile role

GitHub should answer:

1. What does DraconDev make?
2. Where can I use or buy it?
3. Where can I inspect the code if I care?

GitHub should not be the whole business. It is proof and traffic.

### `dracon.uk` role

`dracon.uk` should become the product conversion layer:

- Product pages.
- Pricing.
- Install/download links.
- License pages.
- Changelog.
- Support docs.
- Sponsor/support links.

### YouTube role

YouTube should become the acquisition layer:

- Demo the product.
- Show the problem.
- Link to the product page.
- Link to GitHub for proof.
- Avoid course-first language unless courses become an intentional product.

### GitHub Sponsors role

GitHub Sponsors should be the support layer:

- Visible but not dominant.
- Concrete stats.
- Clear reason to support product development.
- Not framed as the only income path.
