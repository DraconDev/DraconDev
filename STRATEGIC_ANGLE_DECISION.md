# Strategic Angle Decision

**Date:** 2026-06-12  
**Decision:** Choose the product/business maker angle, not the job-seeking angle.

## The question

DraconDev should not chase two horses at once:

1. “I want a good job.”
2. “I want to make money with dev stuff.”

Those angles can coexist, but they should not be equal. If the profile, docs, and funnel try to serve both equally, the visitor has to guess whether DraconDev is a job candidate, a course brand, an open-source maintainer, a freelancer, or a product maker.

The chosen angle is:

> **DraconDev is a maker of usable productized dev assets: Chrome extensions, tools, games, and Rust infrastructure that people can install, use, license, or support.**

A good job can still be a side effect. Strong employers should see the productized work and understand the capability. But employment should not be the primary signal.

## Why this is the better angle

The product/business angle is stronger because it:

- Makes the profile useful to buyers, users, maintainers, employers, and collaborators.
- Turns GitHub from a resume into proof of shipped capability.
- Supports non-employment revenue through products, licenses, subscriptions, and sponsor support.
- Avoids the weak middle ground of “I might be hiring, I might be selling, I might be teaching.”
- Keeps the docs focused on decisions instead of endless audits.
- Lets aligned opportunities come from demonstrated product ownership rather than job-hunt language.

The job angle is not deleted. It is demoted.

## Strategic hierarchy

Use this order:

1. **Product/business first** — sell or distribute usable things.
2. **Proof second** — show code, tests, shipped tools, install pages, and public artifacts.
3. **Aligned opportunities third** — let strong product proof attract better opportunities.
4. **Courses/support last** — use only if they become real products, not default positioning.

## What to stop doing

Stop treating DraconDev like:

- a job application,
- a course catalog,
- a generic open-source portfolio,
- a repo dump,
- an audit archive,
- a freelance-services brochure,
- a “build in public” diary with no product destination.

Stop producing:

- more broad profile audits unless they create a decision,
- more README rewrite variants unless one is adopted,
- more research docs that duplicate existing evidence,
- public product claims without verified install/play pages,
- private repo links as product destinations,
- build-with foundations ahead of usable products.

## What to keep doing

Keep:

- product-first GitHub profile language,
- verified Chrome extension install pages,
- verified game play/install pages when available,
- short public README with product destinations first and code second,
- `dracon.uk` as the product destination,
- YouTube as product/demo distribution,
- GitHub as credibility and proof,
- GitHub Sponsors as support, not the main business,
- build-with foundations as secondary supporting assets.

Keep the existing usable-product-first direction:

> **DraconDev mainly features usable things people can install, open, or play. Build-with foundations are secondary and stay below the product-facing items.**

## Documentation consolidation rule

A doc should exist only if it does one of these jobs:

1. Defines the active direction.
2. Lists product evidence.
3. Drives a near-term execution decision.
4. Serves as the public front door.
5. Records an archive/migration that affects users or code.

If a doc only repeats research, audits, scores, proposals, or old decisions, archive it or link to it from the active docs.

## Positioning language to use

Use:

- “I make usable things people can run or play.”
- “Chrome extensions, games, tools, and Rust infrastructure.”
- “Use or buy the finished product when it is ready.”
- “Inspect public code if you want the proof.”
- “Build-with foundations are secondary.”
- “Private work is noted honestly, not promoted as public product.”
- “GitHub proves the maker; products carry the money path.”

Avoid:

- “Available for work.”
- “Hire me.”
- “Freelance developer.”
- “I teach developers.”
- “Learning journey.”
- “Course catalog.”
- “Full-stack developer looking for opportunities.”
- “I build cool stuff” without a product destination.

## How this affects the profile

The GitHub profile should not ask for employment. It should make the visitor think:

> “This person ships real products and serious systems. If I need this kind of person, I should talk to them.”

That is stronger than:

> “This person wants a job.”

The profile can still help with employment, but indirectly. The direct ask is product/business.

## How this affects the business

The business should focus on repeatable assets:

1. Browser-extension products.
2. Creator tools.
3. Developer infrastructure.
4. Self-hosted licenses.
5. Managed hosting or paid tiers.
6. Sponsor/support as a secondary layer.

Do not make the business depend on hourly work. If service revenue appears, package it as a fixed-scope productized offer tied to an existing asset.

## How this affects the docs

The docs should now answer fewer questions, more clearly:

1. What does DraconDev make?
2. Which products are usable now?
3. Which products are build-with foundations?
4. Which products are private or unverified?
5. Where does money come from?
6. What is the next action?

If a doc does not answer one of those questions, it is probably historical, not active.
